# OWBastion Web Platform

`OWBastion/owbastion.codes` 是 Bastion 生态的 Web Platform 与 operational control plane。
它为玩家、审核人员、管理员、开发者和维护者提供统一的业务数据、审核、授权、变更编排与运行可见性能力。

> **当前状态：目标架构**
>
> 本仓库目前以架构契约和实施方向为主。下文描述的是目标能力，不代表所有模块已经完成或已经具备生产可用性。

## 生态职责

每类事实只保留一个权威来源：

| 仓库 | 权威职责 |
| --- | --- |
| [`OWBastion/Bastion`](https://github.com/OWBastion/Bastion) | 已发布的游戏代码、内容定义、称号源文件、构建、发布版本和公开快照 |
| `OWBastion/owbastion.codes` | 平台业务数据、公共与管理端 Web 应用、审核、发放、平衡草稿和编排 |
| [`OWBastion/qqbot`](https://github.com/OWBastion/qqbot) | QQ 接入、确定性的命令交互和用户通知 |
| [`OWBastion/ocrkit`](https://github.com/OWBastion/ocrkit) | 无状态截图识别和模型生命周期 |

已发布的 Bastion 内容仍以 Bastion Git 和版本化快照为准。平台可以创建草稿、生成变更请求并编排发布，但不能绕过验证、CI、人工审核或发布策略直接修改已发布内容。

## 核心原则

- 一个事实只有一个所有者，避免平台、游戏仓库和缓存之间出现竞争性真相。
- D1 保存平台业务真相；KV 和内存只用于缓存、限流或短期派生结果。
- R2 保存私有截图、OCR 产物、训练候选和大型报告；截图不进入 Git。
- OCR、发放、快照导入和通知等长时间或可重试任务通过队列异步执行。
- 所有副作用都必须幂等、可重试并留下审计记录。
- 公共、玩家私有、审核员、开发者和维护者数据必须有清晰的安全边界。
- API、事件、队列和外部系统契约必须显式版本化。
- 首期采用模块化单体，只有在有明确的扩展性、隔离性或安全需求时才拆分独立服务。

## 产品模块

### 公共门户

面向公众提供标题、事件、地图、挑战、术语、已发布版本、更新日志、公开事件分配报告，以及符合隐私政策的玩家称号、排行榜和地图熟练度查询。

公共页面只能读取已发布快照，不能展示未经批准的草稿数据。

### 玩家区域

认证玩家可以查看关联身份、个人称号和地图进度、截图提交状态、与本人提交相关的 OCR 信息、拒绝原因、重新提交指引、申诉和成长记录。

其他玩家的截图、QQ OpenID、审核笔记和内部风险信号不得暴露。

### 审核与管理后台

审核人员可以处理待审核、失败、低置信度和申诉队列，在原图与 OCR 字段之间进行比对，执行规则评估、字段修正、OCR 重试、批准、拒绝、要求补交或升级处理，并查看不可变审计记录。

人工修正后的样本只有在明确操作后，才能进入训练候选集。

### 开发者控制台

开发者可以检查已发布事件、称号和地图数据，比较版本和快照，模拟事件权重，创建结构化平衡草稿，预览校验结果和差异，并生成可审核的 Bastion Pull Request。

平台不在主 API 运行时执行任意不受信任的 OverPy 或 shell 代码。

## 关键业务流程

### 提交与审核状态机

提交状态必须显式区分审核、发放和游戏发布：

```text
received
→ evidence_pending
→ ocr_pending
→ ocr_processing
→ ready_for_review
→ approved | rejected | resubmission_required
→ grant_pending
→ pr_created
→ granted
→ released
```

异常状态包括 `evidence_failed`、`ocr_failed`、`grant_failed` 和 `notification_failed`。

每次状态转换都要记录允许的前置状态、执行者或服务、时间、幂等行为、审计记录以及重试或回滚路径。

### OCR 编排

平台负责 OCR 编排，QQBot 只负责渠道接入：

```text
创建提交
→ 获取临时附件
→ 校验并将原图写入 R2
→ 计算 SHA-256 和感知哈希
→ 投递 OCR 任务
→ 使用私有对象键调用 OCRKit
→ 保存原始响应、契约版本、模型版本、警告和耗时
→ 评估挑战规则
→ 路由到审核队列
```

OCR 是证据而不是最终事实。原始 OCR 结果不可覆盖；每次重试或模型升级都创建新的 OCR run，人工修正单独保存。

### Challenge Rule Engine

挑战规则应当是确定性、版本化、可测试和数据驱动的。规则至少需要定义稳定的 `challenge_key` 或 `title_key`、兼容游戏版本、所需 OCR 字段或遥测、完成条件、自动化等级、缺失字段处理、拒绝或人工审核策略，以及发布和退役窗口。

自动化等级包括：

- `manual_only`
- `ocr_assisted`
- `auto_eligible`
- `telemetry_required`

模型置信度本身不能直接触发称号发放。自动批准必须有明确政策和经过测量的误报门槛。

### Bastion 变更编排

涉及已发布内容的变更遵循以下流程：

```text
平台变更集
→ 根据导入的 Bastion schema 校验
→ 生成确定性的 patch/request
→ 创建分支和 Pull Request
→ 执行 Bastion CI 与 OverPy 构建
→ 人工审核并合并
→ 发布
→ 导入新的版本化快照
→ 对账平台状态
```

流程必须具备稳定的外部请求 ID、幂等重试、防重复发放、包含证据和审计关联的 PR 描述、最小权限 GitHub 凭证，以及 GitHub 成功但平台回调失败时的对账机制。不得直接推送受保护的发布分支。

### QQBot 边界

平台 API 应支持 QQ 身份绑定或更新、基于稳定 QQ 消息元数据创建提交、查询玩家可见状态和生成通知任务，并能安全处理重复命令。

QQBot 不应理解 OCR、审核规则、称号槽位或 GitHub 工作流。外部 QQ 标识属于私有数据，公共页面只能使用平台安全的玩家标识或显示名称。

## 数据与安全

### 数据存储

- **D1**：用户、外部身份、角色权限、提交、附件元数据、OCR run、人工修正、审核审计、发放任务、进度、平衡草稿、GitHub 关联和通知状态。
- **R2**：原始截图、OCR crop/debug 产物、训练候选、导出数据集和大型报告。对象键应不可变、命名空间清晰，并保留内容哈希。
- **KV**：公共快照、短期会话、限流计数和派生查询缓存。不能作为审核状态、发放记录、身份或玩家资产的唯一来源。

### 角色

| 角色 | 权限范围 |
| --- | --- |
| `visitor` | 公共已发布数据 |
| `player` | 自己的资料、提交、进度和申诉 |
| `reviewer` | 截图审核和字段修正 |
| `content_admin` | 挑战、称号、活动和运营发放 |
| `developer` | 平衡草稿、模拟和 PR 准备 |
| `maintainer` | 发布批准、回滚和角色管理 |

管理端应使用 Cloudflare Access 或等效的强认证方案。授权必须在 API 服务端执行，服务之间使用独立凭证，特权操作写入不可变审计日志，破坏性操作需要重新认证或更强确认，密钥不得进入前端 bundle。

截图和审核数据必须按私有数据保护：校验类型、大小和尺寸，限制对象前缀，执行限流和重复检测，使用短期签名 URL，并防止从外部附件地址触发 SSRF。

## 契约、可观测性与测试

`packages/contracts` 是 API schema、领域事件、队列 payload、Bastion 快照 schema、OCRKit 兼容类型和 QQ 请求类型的统一归属地。数据库迁移应前向安全并提供回滚或修复指引；队列消费者必须兼容重复投递和旧版本 payload。

每个请求和异步任务应携带 correlation ID、执行者或服务身份、操作名、契约版本以及相关的提交或变更集 ID。应监控队列延迟、重试、OCR 成功率与耗时、审核吞吐、审核分歧、发放和 PR 失败率、通知失败、快照新鲜度和审计覆盖率。日志不得包含截图字节、密钥、签名 URL 或不必要的私有标识。

测试采用分层策略：

- 单元测试：规则、状态转换和授权；
- 契约测试：QQBot、OCRKit、Bastion 快照和 GitHub 请求；
- D1 迁移与 repository 测试；
- 队列重复投递和幂等测试；
- 使用假的 R2、OCR、GitHub 和 QQ 客户端进行集成测试；
- 玩家提交、审核批准的端到端测试；
- 权限提升、SSRF、文件校验和私有数据暴露的安全测试；
- 支持的游戏版本之间的快照兼容性测试。

常规单元测试不得依赖线上外部服务。

## 变更策略与交付顺序

重大变更应说明范围、负责人、契约和迁移影响、风险、回滚或对账方案、测试、运行验证和文档更新。

涉及 Bastion 内容的功能按以下阶段上线：

```text
只读
→ 草稿生成
→ dry-run 校验
→ 创建 Pull Request
→ 受控生产使用
```

OCR 自动化按以下阶段推进：

```text
人工基线
→ shadow OCR
→ 审核辅助
→ 有限自动资格
→ 根据指标逐步扩大
```

默认交付顺序：

1. 建立仓库结构、契约、认证和 D1 迁移；
2. 迁移只读标题与事件查询 UI；
3. 导入版本化 Bastion 快照；
4. 实现 QQBot 身份和提交 API；
5. 将截图持久化到 R2，并以 shadow mode 接入 OCRKit；
6. 建立审核工作台和审计模型；
7. 实现发放任务和 Bastion PR 编排；
8. 增加玩家进度、熟练度和排行榜；
9. 增加只读平衡分析；
10. 增加结构化平衡草稿和 PR 创建。

首期不应从任意代码编辑、全自动称号发放或通用可视化 OverPy 编辑器开始。

## 完成标准

一个平台变更只有在以下条件满足后才算完成：

- 每个新增字段都有明确的事实来源；
- 状态转换和幂等行为已定义；
- 授权在服务端执行；
- 公共与私有数据边界有测试；
- 迁移和契约变更有文档；
- 外部失败有重试和对账路径；
- 特权操作会产生审计记录；
- 相关单元、契约和集成测试通过；
- 部署与回滚步骤明确；
- 没有绕过 Bastion Git 或 CI 的路径。
